# AGENTS.md

本文档旨在为所有参与 `Mihosh` 项目开发与协助的 AI 智能体（Agents）及代码助手提供项目级上下文，建立统一的开发规范和共识。

## 1. 项目概况
- **项目名称**: Mihosh
- **项目定位**: 一款现代化的 Mihomo (Clash Meta) 终端控制台（TUI 客户端）。
- **核心功能**: 实时连接监控（抓包级过滤与排查）、节点管理与全量测速验证、控制台实时日志抓取、规则组可视化调试等。

## 2. 核心理念与原则
任何协助代码生成或架构设计的 Agent，**必须**严格遵守用户的全域核心指示：
1. **简洁至上 (KISS)**: 代码以最简单的方式解决问题，坚决避免过度工程和防御性代码。
2. **深度分析 (First Principles)**: 遇到报错或性能瓶颈，从第一性原理出发拆解原因，而非套用模版方案进行临时修补。
3. **事实为本**: 在进行架构迁移或调用外部 API 前，必须基于代码事实（阅读现有文件）。

## 3. 技术架构与选型
- **基础语言**: Go 1.24+
- **CLI/配置**: `spf13/cobra` 和 `spf13/viper`
- **UI 引擎**: `charmbracelet/bubbletea` (Elm 架构)、`bubbles` 和 `lipgloss`
- **开发环境**: Windows 原生环境 (推荐 Powershell 脚本和路径规范)

## 4. 领域驱动目录结构
项目遵循标准化的 Go Layout 和弱化的 DDD 分层管理：
- `cmd/`: CLI 命令行工具入口及命令解析。
- `internal/domain/model/`: 业务实体定义与通用响应结构体。
- `internal/infrastructure/`: 基础设施层适配器（如 HTTP 客户端、WebSocket 双向流封装、本地配置读写）。
- `internal/app/service/`: 业务用例与服务编排层（处理上层指令，调用基础设施层 API）。
- `internal/ui/tui/`: 终端应用渲染层，切分了不同的 Page (页面) 和 Components (组件)。

## 5. 已知优化方向与“雷区”
Agent 介入开发时，请特别注意避免以下问题，并将其作为长期的 Refactor 目标：

### 5.1 Bubbletea 的“上帝模型” (God Object) 问题
**现状 & 规范**：`ui/tui` 下存在包含几十个状态的庞大主模型及臃肿的 `page_updates.go` 生命周期。
**行动指南**：修改 TUI 交互时，应尽量将状态机推向 `pages/` 内部的子组件模型，并通过 `tea.Msg` 进行父子消息传递，**坚决不随意往全局根 Model 堆放游离状态**。

### 5.2 大规模高频并发时的 Goroutine 控制
**现状 & 规范**：节点测速等场景若节点规模大，直接套用 `go func()` 将干穿系统句柄。
**行动指南**：任何针对切片/集合发起的批量并发网络调用（HTTP/TCP），必须通过 Worker Pool、Channel 或 `x/sync/semaphore` 实施并发数硬性约束（如 20 个 Worker），谨守系统资源底线。

### 5.3 内存碎片与长切片拷贝 (GC 炸弹)
**现状 & 规范**：TUI 往往具备实时刷屏和列表追加（如：日志追踪、活跃连接关闭变为历史连接）。
**行动指南**：严禁在超长切片前端插入（即 `append([]T{new}, slices...)`）或在超额时用切片截断抛弃头部（`slices[1:]`）。需要限制固定历史记录（如最近 1000 条）时，务必使用性能优秀的环形缓冲区（**Ring Buffer**）。

### 5.4 WebSocket 的样板代码冗余
**现状 & 规范**：不同通道（Logs, Memory, Connection）存在类似的断线重连、死锁屏蔽逻辑。
**行动指南**：若要新增实时订阅数据源，不要再去 Copy 冗余代码，要求使用 Go 1.18+ 的泛型或函数回调来统一重连生命周期。

## 6. 开发与审查工作流
1. 涉及较复杂的重构任务时，必须先拉取项目全貌并产出 `Implementation Plan` 给人类开发者审核。
2. 保持对中英双语输出的要求：理解需求时进行全局搜索和深度推理；在 `Task List` 和终端回复时始终保证 **全中文** 直观输出。
